Life Cycle of a View

Introduction

In this guide, we're going to to cover the life cycle of the views that Ramen comes with out of the box. This will give you the available hooks you can use to modify their behavior, as well as what you can do in each of the hooks. Mastering hooks is the key to creating complex custom views.

Ramen.view.Box

The box view is nothing more than a managed HTML tag. In most cases, you shouldn't be extending this. However, it is the most basic kind of view. If you just want to inject a raw snippet of HTML into the page and have no need for it to be dynamic, then this is what you want to start with.

All view's follow this same life cycle, throughout this guide only the changes to these hooks will be described:

create

  1. initView

    By the time you get to this hook, your view's config has already been applied, but not much else has happened. In this hook you should be setting up all basic config you need to as well as doing any setup possible prior to rendering such as adding or removing classes.

render

  1. onRenderBefore(view, container, index)

    Prior to this hook, your view has been fully initialized, however no rendering has taken place. The key power of this hook is that by returning false you can stop your view from rendering altogether. This allows you to setup rendering for a later time if your view isn't ready yet.

    Parameter Note
    view the view that the hook belongs to
    container the container render was called with
    index the index render was called with
  2. onRenderDuring

    Not much changes between onRenderBefore and onRenderDuring, however this is a good place to inject render-time functionality.

  3. onRenderAfter

    Prior to this hook firing, the view will have been inserted into it's container, thus fully being rendered. Any sizing and layout that requires your view to be on the page should be done in this hook.

destroy

  1. onDestroyBefore

    Similar to onRenderBefore, this hook gives you the ability to cancel the destruction of your view by returning false.

  2. onDestroy

    This hook is used for doing any cleanup any functionally that you may have added in the other hooks.

Ramen.view.View

The standard view builds on top of box by giving you access to templates, childEls, renderSelectors and domListeners. If your view doesn't need to hold other views, and doesn't deal directly with data stored in models, this is where you want to start.

render

  1. onRenderDuring

    callParent inserts the rendered template, pulls childEls, renderSelectors and sets up domListeners.

    Once callParent is called you can use your rendered html, such as adding styles or classes to childEls or injecting more elements into your template based on complex data. It's better to do as many changes to HTML as possible in this stage so that you avoid a browser repaint or reflow. These can both occur if you wait until onRenderAfter to make your changes.

Ramen.view.binding.BindingView

Binding view builds on view by giving you data bindings that are used to read data from a Ramen.data.Model.

destroy

  1. onDestroy

    callParent destroys all bindings that have been added.

Ramen.view.container.Container

The container view, as it's name suggests, is a view that can hold and render other views. It supports the use of layouts to give you control of how and where each of its children is rendered.

create

  1. initView

    callParent sets up the items dictionary with any child items currently stored in the items parameter.

    Prior to executing callParent the easiest way to setup child items is set the items parameter to an array of item configs. This is also a good time to set any dynamic values needed in itemDefaults.

    After executing callParent the items dictionary will be created, and as such you'll be required to use the standard add, instert and remove calls to modify child items.

destroy

  1. onDestroy

    callParent destroys all items as well as the current layout.

Ramen.view.container.CollectionContainer

  1. initView

    callParent sets up the emptyView and attaches all needed events to the collection. This also overrides the current value of the items parameter, meaning that items cannot be used.
